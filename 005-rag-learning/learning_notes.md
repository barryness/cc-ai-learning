## 1. 起点问题：为什么 ChatGPT 不知道你公司的事？

### 场景

你问 ChatGPT："我们公司年假多少天？"

ChatGPT 回复："作为AI助手，我无法访问您公司的内部政策。建议您查阅员工手册或咨询HR部门。"

为什么？因为 ChatGPT 训练数据里没有你公司的员工手册。

### 三个硬伤

| 问题 | 原因 | 后果 |
|------|------|------|
| 知识过时 | 训练数据有截止日期 | 不知道"今天"的事 |
| 幻觉 | LLM 不会说"不知道"，而是编一个 | 产生看似合理但错误的内容 |
| 没有私有知识 | 训练数据是公开互联网 | 不知道企业内部文档 |

### 怎么解决？

**RAG = 检索 + 增强 + 生成**

```
用户: "我们公司年假多少天？"
         │
         ▼
   ① 检索：从公司文档中搜索相关段落
         → 找到："年假15天，带薪病假7天"
         │
         ▼
   ② 增强：把文档和问题拼接到 Prompt
         → "根据以下文档：[文档1] 办公政策：年假15天... 请回答：年假多少天？"
         │
         ▼
   ③ 生成：LLM 基于增强 Prompt 生成答案
         → "根据公司办公政策，年假为15天。"
```

**一句话：让 LLM 拿着你的文档回答问题，而不是凭记忆瞎猜。**

## 2. RAG 三阶段：检索 → 增强 → 生成

### 阶段一：检索（Retrieval）

**要做的事：** 从知识库里找到和用户问题最相关的文档。

```
用户问题："AI Tutor 用了什么技术栈？"
         │
         ▼
  知识库中有 100 篇文档
         │
         ▼
  检索找出最相关的 3 篇：
  [1] 技术架构 (相关度 92%)
  [2] 产品介绍 (相关度 45%)
  [3] 公司介绍 (相关度 12%)
```

检索质量直接决定 RAG 系统的上限。如果检索没找到相关文档，LLM 再强也答不对。

### 阶段二：增强（Augmentation）

**要做的事：** 把检索到的文档和用户问题拼接成一个"增强 Prompt"。

```
[System Prompt]
你是一个基于内部知识库回答问题的助手。

[检索到的文档 - Context]
[文档1] 技术架构：后端使用FastAPI框架...
[文档2] 产品介绍：AI Tutor支持20+编程语言...

[用户问题]
AI Tutor 用了什么技术栈？

请基于以上知识库内容回答。
```

**关键设计原则：**
- 上下文放在问题前面 → LLM 对靠前的信息关注度更高
- 标注来源 → 用户可以追溯信息出处
- 明确"找不到就说不知道" → 降低幻觉风险

### 阶段三：生成（Generation）

**要做的事：** LLM 看到增强后的 Prompt，生成答案。

```
LLM 看到：
  - System Prompt：你是知识库助手
  - Context：技术架构文档
  - Question：用了什么技术栈？

LLM 生成：
  "根据[文档1]，AI Tutor后端使用了FastAPI框架，
   数据库使用PostgreSQL，前端使用React+TypeScript。"
```

**参数建议：**
- `temperature=0.1~0.3`：事实问答用低温度，减少"创造性发挥"
- `max_tokens=500`：控制答案长度，避免废话

## 3. 为什么需要 Embedding 而不是 TF-IDF？

### TF-IDF 的做法

```
文档向量化：每个词是一个维度，值是 TF-IDF 权重
  "我爱编程" → [TF("我"), TF("爱"), TF("编程"), IDF 修正...]
  
相似度计算：两个向量的余弦相似度
  "我爱编程" vs "我喜欢写代码" → 相似度 ≈ 0（字面完全不同）
```

**致命缺陷：只看字面，不理解意思。**

### Embedding 的做法

```
文档向量化：BERT 模型输出 768 维语义向量
  "我爱编程" → [0.23, -0.15, 0.78, ...]  (768 个数字)
  "我喜欢写代码" → [0.21, -0.13, 0.82, ...]  (768 个数字)
  
相似度计算：
  Cosine("我爱编程", "我喜欢写代码") ≈ 0.85 ✅
```

**核心优势：理解语义，同义词、近义词、上下文都能捕捉。**

### 对比

| | TF-IDF | Embedding (BERT) |
|------|------|------|
| 原理 | 词频统计 | 深度学习语义编码 |
| 向量维度 | 词汇表大小（稀疏） | 768（稠密） |
| "汽车" vs "轿车" | 相似度=0 | 相似度≈0.85 |
| "技术栈"搜"FastAPI" | 找不到 | 能找到 |
| 需要分词 | 是（jieba） | 否（BERT 自带 tokenizer） |
| 速度 | 极快 | 较慢（需 GPU） |

**结论：TF-IDF 适合入门理解原理，生产环境必须用 Embedding。**

## 4. Chunking：为什么要把文档切碎？

### 问题的根源

一篇 50 页的 PDF，如果整篇做一个 Embedding：
- 模型最大输入 512 token → 超长截断
- 整篇语义被"平均"掉 → "技术架构"和"办公政策"混在一起
- 检索精度极差

### 分块的作用

```
原始文档 (5000 字):
┌────────────────────────────────────────────────┐
│ 第一章 公司介绍 ...                               │
│ 第二章 技术架构 ...                               │
│ 第三章 办公政策 ...                               │
└────────────────────────────────────────────────┘

分块后 (每块 500 字, 重叠 50 字):
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  块 1    │ │  块 2    │ │  块 3    │ │  块 4    │
│ 公司介绍  │ │ 技术架构  │ │ 办公政策  │ │    ...    │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
     ↑            ↑
     └── overlap ─┘ (50 字重叠，避免关键信息被切断)
```

### 分块策略

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 固定大小 | 每 500 字切一块 | 通用文档 |
| 语义分句 | 按句号/段落切 | 规范文本 |
| 递归分块 | 先段落 → 太长按句子 → 最后按字数 | 最灵活（推荐） |

### 关键参数

- **chunk_size=500**：每块约 500 字。太小语义不完整，太大检索精度下降
- **chunk_overlap=50**：相邻块重叠 50 字。防止"FastAPI 框架"的"FastAPI"在块1末尾，"框架"在块2开头

**直觉：分块 = 把一本书拆成"相关的段落卡片"，搜索时找到最相关的卡片，而不是翻整本书。**

## 5. 向量数据库：ChromaDB 做了什么？

### 传统数据库 vs 向量数据库

```
MySQL 索引:
  B+ 树 → 按数值排序 → 快速范围查询
  "SELECT * WHERE age BETWEEN 20 AND 30" ✅
  "SELECT * WHERE 语义相似" ❌

ChromaDB 索引:
  HNSW 图 → 按向量距离排序 → 快速最近邻搜索
  "找与 query 向量最接近的 5 个文档" ✅
```

### ChromaDB 的核心抽象

```
Client (客户端)
  └── Collection (集合，相当于 MySQL 的 Table)
       ├── documents (原始文本)
       ├── embeddings (向量，自动计算)
       ├── metadatas (元数据，如标题、来源)
       └── ids (唯一标识)
```

### HNSW 索引：为什么搜索这么快？

```
全量对比（暴力搜索）：
  100 万个向量 × 768 维 → 逐个算距离 → 太慢

HNSW（分层可导航小世界图）：
  顶层：少量节点，快速跳转到大致区域 (高速公路)
  中层：中等节点，缩小范围 (主干道)  
  底层：所有节点，精确查找 (小巷)
  
  复杂度：O(log N) vs 暴力搜索 O(N)
```

**类比：在大城市找人 → 先定位到区（顶层）→ 再定位到街道（中层）→ 找到具体门牌号（底层）**

### 两种存储模式

| 模式 | 代码 | 特点 |
|------|------|------|
| 内存 | `chromadb.Client()` | 进程重启数据丢失，适合 Demo |
| 持久化 | `chromadb.PersistentClient(path="./db")` | 数据存磁盘，重启保留，适合生产 |

## 6. Prompt 增强：怎么把检索结果"喂"给 LLM？

### 核心思路

LLM 的 Context Window 就是你给它的所有信息。把检索到的文档放进去 = 给 LLM "参考资料"。

### Prompt 模板设计

```
┌─────────────────────────────────────┐
│ System Prompt                       │
│ "你是基于知识库的助手，规则：..."      │
├─────────────────────────────────────┤
│ Context (检索结果)                   │
│ [文档1] 标题：技术架构                │
│ 内容：后端使用FastAPI...              │
│                                      │
│ [文档2] 标题：产品介绍                │
│ 内容：月费29元，年费299元...           │
├─────────────────────────────────────┤
│ User Question                       │
│ "技术栈是什么？"                      │
├─────────────────────────────────────┤
│ Instruction                         │
│ "请基于以上知识库内容回答用户问题。"    │
└─────────────────────────────────────┘
```

### 设计要点

1. **System Prompt 放最前面**：定义 LLM 的行为准则
2. **检索结果要标注编号**：`[文档1]` `[文档2]` → LLM 回答时可以引用来源
3. **明确"不知道"的规则**：降低幻觉
4. **Context 在 Question 之前**：LLM 对靠前内容关注度更高（Lost in the Middle 效应）

### 怎么处理检索不到的情况？

```python
if not retrieved_docs:
    return "抱歉，知识库中没有找到相关信息。"
```

不要硬着头皮让 LLM 回答空上下文 → 它会开始编。

## 7. 多轮对话：LLM 怎么"记住"上一句话？

### 问题场景

```
用户: "AI Tutor月费多少钱？"
AI:   "月费29元。"
用户: "那年费呢？"           ← "那年费呢？"本身没有指明是什么产品
AI:   "年费299元。"          ← 需要理解上下文：上一轮在聊 AI Tutor
```

### 如果没有记忆

LLM 看到的是：
```
Message 1: [System Prompt + Context + "那年费呢？"]
```
LLM 完全不知道"那"指的是什么 → 答非所问。

### 加上记忆

LLM 看到的是：
```
Message 1: [System Prompt]
Message 2: [user] "AI Tutor月费多少钱？"
Message 3: [assistant] "月费29元。"
Message 4: [user] [Context + "那年费呢？"]
```
LLM 看到完整的对话历史 → 理解"那"指的是 AI Tutor → 正确回答。

### Token 预算管理

```
Context Window: 每个 LLM 有最大输入长度限制

如果不限制历史:
  第1轮: 500 tokens
  第2轮: 1000 tokens
  ...
  第50轮: 25000 tokens → 超出限制 → 报错

解决方案: max_history_turns=10
  只保留最近 10 轮对话 → 超出后自动丢弃最早的
```

## 8. 混合检索：向量 + 关键词 > 单打独斗

### 为什么需要混合？

```
纯向量检索的问题：
  搜 "API-Key" → 返回"密钥管理系统" ✅ 语义相关
  但可能漏掉标题就叫 "API-Key 配置指南" 的文档
  （Embedding 把 "API-Key" 理解成了"密钥"这个概念）

纯关键词检索的问题：
  搜 "技术栈" → 找不到包含"FastAPI"的文档
  （字面不同，BM25 无法理解它们是同义词）
```

### 混合检索 = 取两者之长

```
用户问题: "API-Key 怎么配置？"
         │
         ├──→ 向量检索 → [密钥管理, 安全配置, 认证鉴权]  ← 语义相关
         │
         └──→ BM25检索 → [API-Key配置指南, API文档]    ← 字面匹配
                  │
                  ▼
            RRF 融合算法
                  │
                  ▼
         [API-Key配置指南, 密钥管理, 安全配置, 认证鉴权, ...]
          ↑ 两路都排前面的 ↑
```

### RRF 算法

**公式：** `score(d) = Σ 1/(k + rank_i(d))`

```
文档A: 向量排第1, BM25排第2 → 1/(60+1) + 1/(60+2) = 0.0164 + 0.0161 = 0.0325
文档B: 向量排第3, BM25排第1 → 1/(60+3) + 1/(60+1) = 0.0159 + 0.0164 = 0.0323
文档C: 向量排第2, BM25排列5 → 1/(60+2) + 1/(60+5) = 0.0161 + 0.0154 = 0.0315

最终排序: A > B > C
```

**直觉：在两路检索中排名都靠前的文档，更可能是真正相关的。**

## 9. 重排序：粗排海选 + 精排决选

### 为什么还需要重排序？

```
向量检索和 BM25 都是 Bi-Encoder（双塔模型）：
  Query → [Encoder] → Query向量 ─┐
                                   ├─→ 点积 → 相似度
  Doc   → [Encoder] → Doc向量  ─┘
  
  优点：快（文档可以预先编码缓存）
  缺点：精度一般（Query 和 Doc 在编码时没有交互）
```

```
Cross-Encoder（交叉编码模型）：
  [Query + Doc] → [Encoder] → 相关度分数
  
  优点：精度高（Query 和 Doc 在模型内部做全注意力交互）
  缺点：慢（每次都要重新计算，不能预缓存）
```

### 两阶段架构（工业标准）

```
粗召回（Bi-Encoder）          精排（Cross-Encoder）
    ┌──────────┐                ┌──────────┐
    │ 10000 篇  │  ──向量+BM25──▶│  Top-20  │  ──Cross-Encoder──▶  Top-3
    │ 文档      │                │  候选     │                    │  最终结果 │
    └──────────┘                └──────────┘                    └──────────┘
      速度快                       速度中等                       返回给用户
      精度低                       精度高
```

**类比：**
- 粗召回 = HR 筛简历（快速过 1000 份 → 挑出 20 份）
- 精排 = 面试官面试（跟 20 个候选人都聊一遍 → 挑出最好的 3 个）

## 12. 生产环境注意事项

### 1. Embedding 模型选择

- `bert-base-chinese`：入门首选，本地缓存离线可用，768 维
- `BAAI/bge-large-zh-v1.5`：中文 SOTA，1024 维，召回率提升 10-15%
- `text2vec-base-chinese`：中文句向量，CoSENT 训练

### 2. 分块大小调优

- 太大：语义被平均，检索不准
- 太小：语义不完整，LLM 得不到足够上下文
- 建议：300-800 字，根据文档类型实测调优

### 3. 检索数量

- `top_k` 太小：可能漏掉相关文档
- `top_k` 太大：Context 超长，LLM 注意力分散
- 建议：3-5 个，粗排用 10-20 个做候选池

### 4. 评估体系

- 用 RAGAS 框架量化评估：忠实度、答案相关性、上下文相关性
- 定期用真实用户问题做回归测试

### 5. 增量更新

- 新文档 → 分块 → Embedding → 插入 ChromaDB
- 不需要重建整个索引

### 6. 安全

- Prompt Injection 风险：文档内容可能包含恶意指令
- 建议：对文档内容做预处理，限制 LLM 的 system prompt 权限

## 总结

**RAG 的本质：**

```
LLM 的知识 = 训练数据（冻结在训练时刻） + Context（每次对话动态注入）
                                              ↑
                                    这就是 RAG 做的事情
```

**记住这张图：**

```
用户问题 → 检索 → 在知识库中找到相关文档 → 拼入 Prompt → LLM 生成答案
   ↑                                                    ↓
   └──────────── 多轮对话记忆 ←──────────────────────────┘
```
