# RAG 系统架构文档

> 从企业知识库场景出发，逐层拆解 RAG 的每个组件。

---

## 一、为什么需要 RAG

### 企业场景的真实痛点

你是一家公司的 CTO。公司有上千份内部文档——产品手册、技术文档、HR 政策、合同模板。员工每天反复问同样的问题：

> "年假怎么申请？" "报销流程是什么？" "这个 API 怎么用？"

你试过：
1. **写 Wiki** → 没人维护，半年后全是过时信息
2. **培训新人** → 成本高，人员流动后知识丢失
3. **直接问 ChatGPT** → 它不知道你公司内部的事情，还会编造答案

### RAG 解决的核心问题

```
┌─────────────────────────────────────────────────────────┐
│                    纯 LLM 的三大硬伤                      │
├─────────────────┬───────────────────────────────────────┤
│ 知识过时         │ 训练数据截止到某个日期，不知道"今天"的事   │
│ 幻觉             │ 遇到不知道的，LLM 会"编"一个看起来像真的答案│
│ 没有私有知识      │ LLM 没见过你公司的内部文档               │
└─────────────────┴───────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│                      RAG 的解决方案                      │
├─────────────────┬───────────────────────────────────────┤
│ 知识过时         │ → 检索最新文档，实时更新                │
│ 幻觉             │ → 给 LLM 提供"参考答案"，约束生成        │
│ 没有私有知识      │ → 接入企业内部文档                     │
└─────────────────┴───────────────────────────────────────┘
```

**一句话：RAG = 先检索（Retrieve）+ 再增强（Augment）+ 最后生成（Generate）。**

## 三、核心组件详解

### 3.1 文档分块（Chunking）

**为什么必须分块：**
- Embedding 模型有最大输入长度（通常 512 token）
- 一篇 50 页的 PDF 作为一个向量 → 语义被"平均"掉，检索精度极差
- 分块后，检索返回的是"相关的段落"，而非整篇文档

**分块策略：**

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 固定大小 | 每 500 字切一块 | 通用文档 |
| 语义分句 | 按句号/段落切 | 规范文本 |
| 滑动窗口 | 每 500 字一块，重叠 50 字 | 避免切断语义 |
| 递归分块 | 先按段落，太长再按句子，再按词 | LangChain 默认方式 |

**关键参数：**
- `chunk_size`：每块大小（推荐 300~500 token，中文约 500~800 字）
- `chunk_overlap`：相邻块重叠量（推荐 10-20%），避免关键信息被切断在边界

```
原始文档: [AAAAAAAAABBBBBBBBBCCCCCCCCCC]
           │  chunk1  │
                │  chunk2  │
                     │  chunk3  │
           ← overlap →  ← overlap →
```

### 3.2 Embedding

**做什么：** 把文本块转成高维向量（如 768 维 float 数组）。

**为什么用 Embedding 而不是 TF-IDF：**
- TF-IDF：只看词频，"汽车"和"轿车"相似度=0（字面不同）
- Embedding：理解语义，"汽车"和"轿车"相似度≈0.85

**中文场景推荐模型：**

| 模型 | 维度 | 特点 |
|------|------|------|
| `bert-base-chinese` | 768 | 通用中文，本地离线可用 |
| `BAAI/bge-large-zh-v1.5` | 1024 | 中文 SOTA，需联网下载 |
| `shibing624/text2vec-base-chinese` | 768 | 中文句向量，CoSENT 训练 |

### 3.3 向量检索（Retrieval）

**做什么：** 在向量数据库中找到与用户问题最相似的文档块。

**流程：**
1. 用户问题 → Embedding → 查询向量
2. 查询向量 → HNSW/ANN 搜索 → 找到最接近的 K 个文档块向量
3. 返回 K 个文档块（通常 K=3~10）

**关键指标：**
- **召回率（Recall）**：相关文档被检索到的比例。越高越好。
- **精确率（Precision）**：检索结果中真正相关的比例。越高越好。

**提高召回率的策略（见 Step 5）：**
- 混合检索（向量 + BM25 关键词）
- 重排序（Re-ranking）：粗召回 → 精排
- Query 改写：把用户问题展开/改写为多个查询

### 3.4 Prompt 增强（Augmentation）

**做什么：** 把检索到的文档块和用户问题拼接成一个"增强 Prompt"。

**Prompt 模板：**
```
[System]
你是一个基于内部知识库回答问题的助手。
规则：
1. 优先使用下面提供的文档内容回答
2. 如果文档中没有相关信息，请明确说"知识库中暂无相关信息"
3. 回答时引用来源

[Context - 检索到的文档]
[文档1] 标题：xxx | 内容：xxx
[文档2] 标题：xxx | 内容：xxx
...

[User Question]
用户问题...

请基于以上知识库内容回答。
```

**关键设计：**
- 引用文档编号，用户可追溯信息来源
- 明确指示"找不到就说不知道"——降低幻觉
- Context 放在 Question 前面——LLM 对靠前的内容关注度更高

### 3.5 生成（Generation）

**做什么：** LLM 基于增强后的 Prompt 生成最终答案。

**参数建议：**
- `temperature=0.1~0.3`：低温度减少幻觉，事实问答不要高温度
- `max_tokens=500~1000`：根据答案长度需求调整
- `stream=True`：用户体验更好（逐字输出）

## 五、技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 文档解析 | PyMuPDF (fitz) | 中文 PDF 提取最稳定 |
| 分词 | jieba | 中文分词，轻量 |
| Embedding | bert-base-chinese | 本地缓存，离线可用 |
| 向量数据库 | ChromaDB | 轻量，pip install 即可 |
| LLM | DeepSeek Chat | 中文好，便宜 |
| 关键词检索 | BM25 (rank-bm25) | 比 TF-IDF 更精准的词汇匹配 |

## 六、性能基线

| 指标 | Step1 | Step5 |
|------|-------|-------|
| 检索方式 | TF-IDF | 向量 + BM25 + 重排序 |
| 召回率@5 | ~40% | ~85% |
| 单次查询延迟 | <50ms | <200ms |
| 支持的文档量 | ~100 篇 | ~10000 篇 |
| 持久化 | 无 | ChromaDB 持久化 |
