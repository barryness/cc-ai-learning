# 练习2：理解 top_k 的影响

## 结论

**top_k 的效果取决于问题类型**：

| 问题类型 | top_k 影响 | 原因 |
|----------|-----------|------|
| 单文档可回答 | 几乎看不出 | 第1名文档就包含全部答案 |
| 跨文档问题 | **非常明显** | 答案分散在多篇文档，top_k 太小会漏掉 |

你感觉"看不出效果"是因为之前测试的问题都是单文档问题——正确答案排第1，top_k=1 就够了。

## 测试代码：跨文档问题 × 不同 top_k

### test_topk_cross_doc.py

```python
"""
测试 top_k 对跨文档问题的真实影响。

核心场景：答案分散在"技术架构"和"产品介绍"两篇文档中，
但 TF-IDF 检索排名中"产品介绍"排在第5名。
→ top_k < 5 时，LLM 根本看不到产品信息，必然回答不全。
"""
from minimal_rag import RAGPipeline, DOCUMENTS

rag = RAGPipeline(DOCUMENTS)
query = '公司的技术栈和产品价格分别是什么？'

# 先看检索排名
print("=== 检索排名（top_k=10） ===")
docs = rag.retriever.retrieve(query, top_k=10)
for i, doc in enumerate(docs, 1):
    marker = ""
    if '技术' in doc['title']:
        marker = " ← 技术栈解答"
    if '产品' in doc['title']:
        marker = " ← 价格解答"
    print(f"  第{i}名: {doc['title']} ({doc['score']}){marker}")

print()

# 对比不同 top_k
for k in [2, 3, 5]:
    print(f"\n{'='*50}")
    print(f"top_k = {k}")
    print(f"{'='*50}")
    docs = rag.retriever.retrieve(query, top_k=k)
    print(f"检索到的文档: {[d['title'] for d in docs]}")
    answer = rag.ask(query, top_k=k, verbose=False)

    has_price = '29' in answer or '299' in answer
    has_tech = 'FastAPI' in answer or 'PostgreSQL' in answer
    print(f"技术栈: {'✅' if has_tech else '❌'} | 价格: {'✅' if has_price else '❌'}")
```

## 分析

### 为什么 top_k=2 两个都答不了？

第1名"大数据实时处理工具"虽然和查询有共同词（"技术"、"使用"），但内容和公司无关。第2名"公司介绍"也没提技术栈细节。LLM 拿到这两篇文档，只能如实说"不知道"——这就是 RAG 正确行为，没有幻觉。

### 为什么第1名排错了？

```
查询: "公司的技术栈和产品价格分别是什么？"
                                 ↑      ↑
第1名文档: "目前大数据实时处理技术使用的工具" → 和"技术"、"使用"、"工具"都有字面重叠
第3名文档: "技术架构" → 才真正是技术栈信息
```

TF-IDF 是字面匹配，不是语义理解。它不知道"大数据技术工具"和"公司技术架构"是两回事。这就是为什么工业级要用 Embedding 模型——语义搜索不会犯这种错误。

### top_k 太大有什么问题？

```
top_k=5 时检索到了"融资情况"——和问题完全无关。
无关文档会：
  1. 浪费 token（多花钱）
  2. 可能干扰 LLM 判断（噪音）
  3. 挤占上下文窗口
```

## top_k 选择原则

| 场景 | 建议 top_k |
|------|-----------|
| 简单事实查询 | 1-3 |
| 需要推理/对比的问题 | 3-5 |
| 需要综合多文档信息 | 5-10 |
| 知识库文档质量不确定 | 偏大，配合相似度阈值过滤 |

## 为什么你之前的测试"看不出效果"？

之前的问题是：
- "公司混合办公的政策是什么？" → 第1名就是"办公政策"，top_k=1 足够
- "公司融资了多少钱？" → 第1名就是"融资情况"，top_k=1 足够

这些**单文档问题**，答案在第1名文档里就已经完整了，加再多 top_k 也是引入无关文档，不会改变答案。

**检验方法**：找一个"答案分散在多篇文档"的跨文档问题，top_k 的效果就非常明显。

## 深层理解

top_k 本质上是**召回率 vs 精确率**的权衡：
- top_k 小 → 精确率高（返回的都是最相关的），召回率低（可能漏掉相关文档）
- top_k 大 → 召回率高（不太会漏），精确率低（引入噪音）

RAG 里通常偏向"多召回"——LLM 能自己忽略噪音，但漏掉的文档 LLM 无法凭空补回来。
