# 练习3：替换检索算法 —— 余弦相似度 → 欧氏距离

## 现象

把 `cosine_similarity` 换成 `euclidean_distances` 后，所有查询都检索不到正确文档。

## 根因：方向反了

两种算法的含义完全不同：

| 算法 | 含义 | 判断标准 |
|------|------|---------|
| 余弦相似度 | 向量夹角 | **越大** = 越相关，范围 [0, 1] |
| 欧氏距离 | 向量空间距离 | **越小** = 越相关，范围 [0, ∞) |

代码里的排序逻辑是"分数从高到低"：

```python
top_indices = similarities.argsort()[::-1][:top_k]  # argsort默认升序，[::-1]反转=降序
```

这个降序逻辑对余弦相似度是对的（1.0 > 0.5 > 0.0），但用到欧氏距离上就反了——距离最大的被当成最相关的。

## 测试代码

### test_euclidean_vs_cosine.py

```python
"""验证欧氏距离与余弦相似度的排序方向差异"""
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity, euclidean_distances

# 3篇文档 + 1个查询
query = np.array([[0.5, 0.3, 0.0]])
docs = np.array([
    [0.5, 0.3, 0.0],   # 文档A：和查询完全一样
    [0.4, 0.2, 0.1],   # 文档B：比较像
    [0.0, 0.0, 1.0],   # 文档C：完全不同
])

cos_sim = cosine_similarity(query, docs)[0]
euc_dist = euclidean_distances(query, docs)[0]

print("=== 余弦相似度 ===")
print(f"原始值: {cos_sim}")
print(f"降序排: A(1.0) > B(0.97) > C(0.0)")
print(f"→ 正确：A最相关排第一 ✅")
print()

print("=== 欧氏距离 ===")
print(f"原始值: {euc_dist}")
print(f"降序排: C({euc_dist[2]:.2f}) > B({euc_dist[1]:.2f}) > A({euc_dist[0]:.2f})")
print(f"→ 错误：最不相关的C排第一 ❌")
print()

# 正确做法：距离转相似度
print("=== 正确做法：距离转相似度 ===")
# 方法1: 取倒数（距离为0时会有问题）
# 方法2: 1/(1+d) 把距离映射到 (0, 1]
euc_sim = 1.0 / (1.0 + euc_dist)
print(f"1/(1+距离): {euc_sim}")
print(f"降序排: A({euc_sim[0]:.4f}) > B({euc_sim[1]:.4f}) > C({euc_sim[2]:.4f})")
print(f"→ 正确 ✅")
```

## 为什么"所有答案都找不到了"？

两步叠加导致：

1. **排序反了**：最相关的文档排到了最后，top_k 取的是最不相关的文档
2. **LLM 拿到噪音**：Prompt 里塞的全是不相关文档，LLM 根据规则如实回答"知识库中暂无相关信息"

这是正确的 RAG 行为——拿到噪音文档时不应该瞎编。

## 修复方案

如果要用欧氏距离，需要加一步转换：

```python
# 修改前（错误）
similarities = euclidean_distances(query_vector, self.doc_vectors)[0]

# 修改后（正确）
distances = euclidean_distances(query_vector, self.doc_vectors)[0]
similarities = 1.0 / (1.0 + distances)  # 距离 → 相似度，越大越好
```

## 对比总结

| 算法 | 公式 | 优点 | 缺点 |
|------|------|------|------|
| 余弦相似度 | cos(θ) = A·B / (∥A∥∥B∥) | 只关心方向，不受向量长度影响 | 对零向量无定义 |
| 欧氏距离 | √Σ(Ai - Bi)² | 直观、计算快 | 受向量长度影响大，高维效果差 |
| 点积 | A·B | 最快 | 没归一化时长度影响大 |

**RAG 里为什么普遍用余弦相似度？**

因为 Embedding 向量的**方向**代表语义、**长度**往往代表"信息密度"（长文档向量更长）。余弦只看方向，等价于做了隐式归一化，更适合语义比较。欧氏距离会惩罚"方向相同但长度不同"的向量——同一篇文档的两个不同长度的分块，可能欧氏距离很远但余弦距离很近。

## 关键收获

- 任何检索算法都要明确**分数方向**：越大越好还是越小越好
- 不能只换算法不改排序逻辑
- 余弦相似度在 RAG 里是标配，不是随便选的
