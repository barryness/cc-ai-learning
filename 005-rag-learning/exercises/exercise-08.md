# 练习8：RAG 的局限性

## 1. 什么情况下 RAG 会失效？（至少 3 种场景）

### 场景一：检索不到相关文档

这才是 RAG 最常见的失效模式。原因包括：
- **分词/Embedding 不匹配**（我们反复遇到的 jieba 分词问题）
- **知识库没有答案**（问"竞争对手"就是这种情况——RAG 正确拒绝了）
- **查询和文档的"语言"不同**（用户口语 vs 文档书面语，"咋请假" vs "休假申请流程"）

### 场景二：检索到了文档但 LLM 没用对

- **文档太多、信息过载**：LLM 上下文窗口有限，相关文档排在第 10 名以后就看不到了
- **文档之间信息矛盾**：两篇文档说法不一，LLM 不知道该信谁
- **LLM 忽略文档**：有时候 LLM 凭借训练记忆回答，忽略了提供的文档内容

### 场景三：多跳推理（Multi-hop Reasoning）

RAG 检索一次→生成一次的模式，无法处理需要"查 A → 从 A 得到关键词 → 再查 B → 综合回答"的问题。

```
Q: "公司CTO之前在哪个公司工作？"
  → 需要先检索"公司CTO是谁" → 得到答案"张老师"
  → 再用"张老师"检索 → 找到"曾在Google和微软工作"
  → 综合回答

一次检索只能完成第一步，后面的推理链断了。
```

解决方案：**多跳 RAG**（Multi-hop RAG），迭代检索+生成。

### 场景四：需要结构化答案

RAG 擅长生成自然语言，但不擅长生成精确的结构化数据：

```
Q: "列出所有员工的入职日期和薪资"
  → RAG 可能漏掉一些人、格式不一致
  → 应该直接查数据库，不是 RAG
```

## 2. 检索到的文档不相关怎么办？

这是一个分层处理的问题：

### 第一层：检索时过滤

```python
# 设置相似度阈值，过滤低分文档
SIMILARITY_THRESHOLD = 0.15  # 低于此值直接丢弃
filtered_docs = [doc for doc in results if doc["score"] > SIMILARITY_THRESHOLD]
```

### 第二层：Prompt 约束

```
"如果提供的文档内容无法回答用户问题，请明确告知'当前知识库无法回答此问题'，
 不要编造信息。"
```

这是我们 Demo 里已经做的事情——#8 "公司有哪些竞争对手"正确回答了"暂无信息"。

### 第三层：后处理校验

- 检查生成答案中引用的实体是否在检索文档中出现过
- 如果引用了文档里不存在的数字/人名 → 可能是幻觉 → 标记或丢弃

### 第四层：人工兜底

- 关键业务场景（如医疗/法律）→ 答案标记为"待人工审核"

## 3. 如果知识库有百万级文档，检索怎么加速？

| 规模 | 方案 |
|------|------|
| < 1 万 | 暴力搜索也可以 |
| 1 万 - 100 万 | **近似最近邻（ANN）索引**：HNSW、IVF |
| > 100 万 | **分布式向量数据库** + **多级检索** |

### 多级检索架构（工业级标配）

```
百万文档 ──→ 第1级：粗排（ANN 索引）     → 召回 1000 条
                 ↓
            第2级：精排（高精度 Embedding） → 保留 100 条
                 ↓
            第3级：重排序（Cross-encoder）   → 返回 top 10
```

第 1 级用 HNSW（Hierarchical Navigable Small World）图索引，log(N) 复杂度。这是所有向量数据库（Milvus、Pinecone、ChromaDB）底层做的事。

### 关键词索引加速

```
BM25 也无法百万级暴力搜索 → 用倒排索引（Elasticsearch/OpenSearch）
                          → 先查倒排 → 取 top-N → BM25 精算
```

## 4. RAG 和 Fine-tuning 各适合什么场景？

| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 知识更新 | 改文档即可，秒级生效 | 需要重新训练，小时/天级 |
| 成本 | 检索+推理，按 token 付费 | 训练一次贵，推理便宜 |
| 适合 | 知识密集型、频繁更新的内容 | 风格/格式/推理模式 |
| 可控性 | 可以检查检索到的文档 | 模型内部黑盒 |
| 典型场景 | 客服、文档问答、新闻摘要 | 代码补全、特定写作风格、领域推理 |

### 实际产品中两者往往组合使用

```
RAG 提供最新知识 + Fine-tuning 让模型学会如何利用这些知识

例：法律 AI 助手
  → Fine-tuning：让模型学会"法律三段论"推理模式
  → RAG：注入最新的法律法规条文
```

### 什么时候选 Fine-tuning 而不是 RAG？

- 需要模型学会一种**新的推理方式**（不只是新的知识）
- 需要稳定的**格式输出**（如严格的 JSON schema）
- **延迟敏感**场景——省去检索步骤
- 有大量**标注好的高质量数据**

### 什么时候选 RAG 而不是 Fine-tuning？

- 知识**更新频繁**（每天都有新文档）
- 需要**可解释性**（可以展示参考了哪篇文档）
- **快速上线**（不需要收集标注数据）
- 知识量**远超模型参数量**（海量文档不可能都训练进去）

## 练习过程中的真实体会

经过 8 个练习，实际遇到的 RAG 问题：

| 问题 | 根因 | 属于哪类局限 |
|------|------|------------|
| 5题中4题搜不到 | jieba 中文分词 | 检索层（场景一） |
| "vibe coding"搜不到 | 标题未索引 | 检索字段设计 |
| top_k 看不出效果 | 单文档问题 | 评估方法 |
| 欧氏距离答案全灭 | 排序方向反了 | 工程实现 |
| "年假"搜不到 | jieba 词典不全 | 检索层（场景一） |

五个问题中有三个都是**检索层**的——检索才是 RAG 系统的瓶颈，生成层的 LLM 已经很成熟了。这也是为什么 RAG 研究的重点一直在"如何更准确地检索"上。
