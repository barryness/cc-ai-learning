## 1. 什么是 LLM

### 通俗解释

LLM = Large Language Model = **大型语言模型**。

拆开看：
- **Large**：大。参数多、数据多、计算多
- **Language**：语言。它学习的材料是"人类写的所有文字"
- **Model**：模型。它是一个数学模型，不是数据库，不是搜索引擎

它的本质是一个**超级版的"输入法联想"**。你用手机输入法打字时，输入法会预测下一个词。LLM 做的就是这件事——但它看得更远、理解更深、规模大了几万倍。

**LLM 不是"知道答案"，而是"擅长续写"**。

### 技术解释

LLM 是一个**神经网络模型**，它的训练目标极其简单：

> **给定前文，预测下一个 Token**

这个目标叫做 **Next Token Prediction（下一个词预测）**。

训练过程：
1. 喂入海量文本（整个互联网、书籍、论文）
2. 每次都遮住最后一个词，让模型猜
3. 猜错了就调整参数（反向传播）
4. 重复几万亿次

最终，模型学到的不是"背答案"，而是**语言的统计规律**——词与词之间、概念与概念之间的概率关系。

```
输入："中国的首都是____"    → 模型输出概率：[北京: 0.92, 南京: 0.03, ...]
输入："1 + 1 = ____"        → 模型输出概率：[2: 0.95, 3: 0.02, ...]
```

这就是全部。没有"理解"，没有"意识"，只有**概率**。

### 实际案例

想象你有一个巨大的哈希表，key 是前文，value 是所有可能的下一个词及其概率。LLM 就是这张表的"智能压缩版"——它不存储原文，而是存储了"生成这些文本的规则"。

### 实际体验

用 `demo.py` 中的 `demo_llm_basics()` 可以亲自体验：给出一段文本，观察 LLM 如何"续写"，以及同样的输入为什么每次输出可能不同（采样策略）。

## 2. Token 是什么

### 通俗解释

Token 是 LLM 的"基本单位"。不是字，不是词，而是**介于字和词之间的东西**。

把 Token 理解为**积木块**：
- 英文：一个词 ≈ 1 个 Token（"apple" = 1 个 Token）
- 中文：一个字可能 1 个 Token，也可能一个字拆成多个 Token
- 特殊符号：一个符号 = 1 个 Token

LLM 不看"字"，也不看"字母"，它看的是 **Token ID**。

```
"Hello, world!" → [15496, 11, 995, 0]  ← Token ID 列表
```

### 技术解释

Tokenization（分词）是把文本转成 Token ID 的过程。用的工具叫 **Tokenizer**（分词器）。

OpenAI 的 Tokenizer（基于 **BPE，Byte Pair Encoding**）：
1. 把所有字符拆成最小的字节
2. 统计最常见的相邻字节对，合并成一个新 Token
3. 重复合并，直到达到预设的词汇表大小（如 100k 个 Token）

```
"低" → Token ID 23102
"成本" → Token ID 40598
"低成本" → Token ID 23102 + 40598（不是 1 个 Token）
```

BPE 的精妙之处：
- 常见词 → 1 个 Token（"the"、"is"、"and"）
- 罕见词 → 拆成多个子词 Token（"antidisestablishment" → 拆开）
- 中文：每个字符基本是 1 个 Token，但有些常用词会合并

### 实际案例

| 文本 | Token 数 | 说明 |
|------|---------|------|
| "Hello, world!" | 4 | 每个词 + 标点 |
| "中国的首都是北京" | 7 | 中文字符 |
| 一首 100 字的诗 | ~100 | 中文约 1 字 1 Token |
| 这篇笔记 | ~3000 | 估算 |

### Token 的实用意义

- **计费**：按 Token 计费（输入 + 输出）
- **限制**：Context Window 限制的是 Token 数，不是字数
- **成本**：中文约 1.5 字/Token，英文约 0.75 词/Token
- **溢出**：当 Token 数超过 Context Window，最早的部分被丢弃

`demo.py` 中的 `demo_tokenization()` 可以实际查看任何文本的 Token 化结果。

## 3. Context Window 是什么

### 通俗解释

Context Window = **上下文窗口** = LLM 一次能"看到"的最大 Token 数。

就像你的短期记忆。你能记住刚才几句话说的内容，但如果聊了一个小时，前面的细节就模糊了。LLM 也一样——它有一个固定的"视野范围"。

```
[Context Window = 4K Tokens]
┌──────────────────────────────────────────────┐
│  开头部分 → ... → ... → 最新部分              │
│  ↑                                        ↑  │
│  最早的内容（可能被丢弃）                  当前输入 │
└──────────────────────────────────────────────┘
```

### 技术解释

Transformer 的 **Self-Attention 的计算复杂度是 O(n²)**：
- n 是 Token 数
- 4K Tokens：约 1600 万次计算
- 8K Tokens：约 6400 万次计算
- 128K Tokens：约 160 亿次计算

这就是为什么 Context Window 不是无限大——**计算量随窗口大小平方级增长**。

不同模型的 Context Window：

| 模型 | 窗口大小 | 约当字数（中文） |
|------|---------|----------------|
| GPT-3.5 | 4K | ~3000 字 |
| GPT-4 | 8K / 32K / 128K | ~6000 / ~24000 / ~96000 字 |
| Claude 3 | 200K | ~15 万字 |
| Gemini 1.5 | 1M / 2M | ~75 万 / ~150 万字 |

### 实际影响

当输入超出 Context Window 时：
1. **最前面的 Token 被丢弃**（从中间开始反而会丢两端）
2. 模型"忘记"了最早的信息
3. 长文档对话时，中间的细节最容易丢失

### 实验

`demo.py` 中的 `demo_context_window()` 演示了以下场景：
- 输入一段超长文本
- 观察当超出窗口时，最早的信息如何"消失"
- 对比不同长度输入的回复质量

## 4. Transformer 是什么

### 通俗解释

Transformer 是一种**神经网络架构**，2017 年由 Google 提出。它是 ChatGPT 和所有现代 LLM 的"地基"。

把 Transformer 想象成一个 **流水线工厂**：

```
输入句子 → [分词] → [编码器] → [注意力层] → [解码器] → 输出
                                    ↓
                            每个词都在"关注"其他词
```

名字的由来：它能把输入序列"变形"（Transform）成输出序列。

### 技术解释（不用公式）

Transformer 的核心创新：**去掉循环（RNN），只用注意力（Attention）**。

之前的模型（RNN/LSTM）是一个词一个词按顺序处理：
```
我 → 爱 → 你
         ↑
        RNN 必须记住前面说了什么（顺序处理）
```

Transformer 是**并行处理所有词**：
```
我 ────→
爱 ────→ Attention（同时看所有词） → 输出
你 ────→
```

Transformer 的主要组件：
1. **Embedding（嵌入）**：把 Token ID 转成向量（一串数字）
2. **Self-Attention（自注意力）**：每个词"看"所有其他词
3. **FFN（前馈网络）**：对每个词独立做非线性变换
4. **Layer Norm（层归一化）**：稳定训练过程
5. **Residual Connection（残差连接）**：防止梯度消失

### 为什么 Transformer 赢了

| 特性 | RNN（旧） | Transformer（新） |
|------|----------|-----------------|
| 处理方式 | 顺序处理 | 并行处理 |
| 训练速度 | 慢 | 快（可 GPU 并行） |
| 长距离依赖 | 难记住 | 自注意力直接连接 |
| 扩展性 | 差 | 好（堆更多层） |

### 简单类比

Transformer 像 **Git 的 diff**：
- RNN 像顺序读代码——从头看到尾，看到后面时前面的忘了
- Transformer 像一次看整个 diff——每个改动都和所有其他改动对比

`demo.py` 中的 `demo_transformer_architecture()` 用代码可视化 Transformer 的结构。

## 5. Attention 为什么重要

### 通俗解释

Attention = **注意力机制** = 让模型知道"该看哪里"。

当你读这句话时：
> "2020 年他在北京吃了烤鸭，**那家店**在王府井。"

读到"那家店"时，你的大脑自动看向"北京"、"烤鸭"、"王府井"。你不需要逐字回忆——你的注意力自然地跳到了关键信息上。

Attention 做的就是这件事。它让每个词都能"看向"所有其他词，并**计算"我该多关注谁"**。

### 核心直觉

Attention 的本质是三个向量之间的运算：

- **Query（查询）**：当前词在问"谁和我相关？"
- **Key（键）**：其他词在说"我包含什么信息"  
- **Value（值）**：如果匹配上了，贡献什么内容

```
Query("那家店") × Key("北京") = 匹配分数 0.2
Query("那家店") × Key("烤鸭") = 匹配分数 0.8  ← 最相关
Query("那家店") × Key("王府井") = 匹配分数 0.6

结果 = 0.2 × Value("北京") + 0.8 × Value("烤鸭") + 0.6 × Value("王府井")
     = 融合了上下文信息的新"那家店"表示
```

### 技术解释

Attention 的计算步骤（简化版）：

```
1. 每个 Token 生成 Q、K、V 三个向量
   Q = Token × Wq（查询矩阵）
   K = Token × Wk（键矩阵）
   V = Token × Wv（值矩阵）

2. 计算注意力分数（Q × K^T）
   分数矩阵[i][j] = Token_i 对 Token_j 的关注度

3. Softmax 归一化
   把分数转成概率分布（和为 1）

4. 加权求和（分数 × V）
   每个 Token 的新表示 = 所有 Token 的 V 的加权和
```

这就是 **Scaled Dot-Product Attention（缩放点积注意力）**。

### 实际案例

用代码看 Attention 比公式直观得多：

```python
import numpy as np

# 假设 3 个 Token，每个 4 维
tokens = np.array([
    [1, 0, 1, 0],  # "我"
    [0, 1, 0, 1],  # "爱"
    [1, 1, 0, 0],  # "你"
])

# 简化版 Attention: Q × K^T
scores = tokens @ tokens.T  # 每个 Token 对其他 Token 的点积

# Softmax
def softmax(x):
    e_x = np.exp(x - np.max(x, axis=-1, keepdims=True))
    return e_x / e_x.sum(axis=-1, keepdims=True)

attention_weights = softmax(scores)

print("注意力权重矩阵（行=查询Token，列=被关注Token）:")
print(attention_weights)
# 每一行表示：当前 Token 对其他 Token 的关注度分布
```

### 为什么 Attention 是革命性的

在 Attention 之前，模型处理长文本靠"记忆"（RNN 的隐藏状态）。
Attention 之后，模型可以直接"查找"（所有位置两两可见）。

好比：
- **RNN** = 用人脑从头到尾记住一篇 10 万字的论文
- **Attention** = 带着全文搜索功能，随时 Ctrl+F

`demo.py` 中的 `demo_attention_mechanism()` 完整演示了这个过程。

## 6. 为什么会产生幻觉

### 通俗解释

幻觉（Hallucination）= LLM **编造看起来合理但实际错误的内容**。

原因很直接：**LLM 的目标是"生成最可能的下一段话"，不是"说出事实"**。

```
用户问："爱因斯坦什么时候访问过中国？"
        ↑ 爱因斯坦从未访问过中国

LLM 的思维过程：
  "爱因斯坦" + "访问" + "中国" → 这几个词经常在一起出现
  → "1922年"（爱因斯坦确实在1922年去过日本）
  → "他去了上海"（编造细节）
  → 输出：一段看起来合理但完全虚构的"历史"
```

### 技术解释

幻觉的根源：

1. **训练目标 = 概率拟合，不是事实验证**
   - LLM 训练时从没见过"对错标签"
   - 它的唯一目标是"输出和训练数据中的续写模式一致"

2. **知识截止 = 没有实时信息**
   - 训练数据停在某个时间点
   - 超出这个时间点的问题，模型只能"编"

3. **压缩损失 = 细节必然丢失**
   - 模型把几 TB 的文本压缩成几百 GB 的参数
   - 不常见的细节被"压缩掉了"，模型用常见模式填补

4. **采样随机性 = 每次输出不同**
   - 模型不是每次都选概率最高的词
   - 采样（temperature > 0）引入了随机性，有时选了"错"的路径

### 幻觉类型

| 类型 | 例子 | 原因 |
|------|------|------|
| 事实幻觉 | "爱因斯坦访问过中国" | 没见过正确数据 |
| 引用幻觉 | 编造论文标题和作者 | 压缩损失 |
| 逻辑幻觉 | "5 > 10，所以..." | 推理能力不足 |
| 指令幻觉 | 没被要求却主动做了 | 过度对齐偏差 |

### 实际案例

```python
# 这会导致 LLM 编造内容
prompt = "请引用一篇 2025 年关于 LLM 幻觉研究的论文"
# LLM 可能会编造一篇不存在的论文，因为训练数据截止更早

# 这就不会
prompt = "在 LLM 中，幻觉的主要原因是什么？"
# LLM 可以基于训练数据中的知识合理回答
```

`demo.py` 中的 `demo_hallucination()` 展示了 LLM 在面对"知识边界外"的问题时如何编造答案。

## 7. 为什么长上下文会失忆

### 通俗解释

"失忆" = LLM 在处理超长文本时，**忘记或混淆了之前的内容**。

这不是 Bug，是**Attention 的固有限制**：

```
文本长度 → 注意力分数被稀释 → 每个词"关注"太多东西 → 关注度趋同 → 等于没关注
```

想象你在一个万人体育场找朋友。如果体育场只有 10 个人，你一眼就能找到。如果有 10000 个人，每个人的脸都模糊了。

同样的道理：当 Context Window 从 4K 扩大到 128K，每个 Token 要"关注" 32 倍多的其他 Token，注意力自然被稀释了。

### 技术解释

长上下文失忆的 4 个原因：

1. **注意力稀释（Attention Dilution）**
   - Softmax 把注意力分数归一化成概率分布
   - Token 越多，每个位置分到的概率越小
   - 128K Tokens 时，每个位置的关注度不到 4K 时的 1/32

2. **位置编码衰减（Position Encoding Decay）**
   - 早期模型（如 GPT-3.5）使用绝对位置编码
   - 距离越远，位置编码越"模糊"
   - 模型难以区分"第 100 句话"和"第 1000 句话"

3. **中间信息丢失（Lost in the Middle）**
   - 研究表明：**LLM 对开头的信息和结尾的信息记得最清**
   - 中间的信息最容易丢失
   - 像人类的序列位置效应（首因效应 + 近因效应）

4. **KV Cache 压力**
   - 生成每个 Token 时，需要缓存所有之前 Token 的 K、V 向量
   - 窗口越大，KV Cache 越大（128K 时可能超过 GPU 显存）
   - 实际实现中可能被迫降级（量化、压缩）

### Lost in the Middle 实验

```
输入：[文档开头] ... [关键信息在中间] ... [文档结尾]
       ↑ 记得清           ↑ 记不清           ↑ 记得清

输入：[关键信息放在开头] ... [文档中间] ... [文档结尾]
       ↑ 记得清                 ↑ 仍然记得
```

关键发现：**把最重要的信息放在 Prompt 的开头或结尾**。

### 实际策略

- 长文档处理：把关键信息放在 Prompt 的开头或结尾
- 分段处理：把长文本切段，分别总结，再汇总
- RAG：不把整份文档塞进 Context，只检索相关片段
- 注意 Token 计数：200K 窗口 ≠ 200K 都有效

`demo.py` 中的 `demo_long_context_loss()` 模拟了注意力稀释过程。

## 8. 为什么 Prompt 会影响结果

### 通俗解释

Prompt = **你给 LLM 的输入文本**。它决定了 LLM "从哪里开始续写"。

LLM 是"续写机器"。你给的 Prompt 就是**起点**。起点不同，续写方向天差地别。

```
Prompt A: "Python 是一种..."
→ 续写：编程语言，由 Guido van Rossum 创建...

Prompt B: "Python 是一种蛇，它..."
→ 续写：生活在热带地区，体长可达...

Prompt C: "写一首关于 Python 的诗"
→ 续写：在代码的世界里，有一条蛇，名叫 Python...
```

同一个模型，三个 Prompt，三种完全不同的输出。模型没变，变的是"起点"。

### 技术解释

Prompt 通过以下机制影响输出：

1. **条件概率分布偏移**
   - LLM 计算 P(next_token | prompt + previous_tokens)
   - Prompt 决定了条件概率的"初始状态"
   - 不同的 Prompt 把概率分布"推向"不同的方向

2. **上下文学习（In-Context Learning）**
   - 在 Prompt 中给例子 → 模型会"模仿"例子模式
   - 不需要修改模型参数，仅靠 Prompt 就能"教"模型做事
   - 原因：Attention 让模型看到了例子的格式、风格、结构

3. **系统提示（System Prompt）的影响**
   - System Prompt = 给 LLM 的"角色设定"和"行为规则"
   - 它叠加在对话的最开始，影响每一轮输出
   - 相当于"给模型戴了一副有色眼镜"

### Prompt 的关键要素

| 要素 | 作用 | 示例 |
|------|------|------|
| 角色设定 | 定义行为模式 | "你是一个资深的 Python 工程师" |
| 任务描述 | 明确目标 | "请重构以下代码" |
| 格式要求 | 控制输出结构 | "用 JSON 格式返回" |
| 例子（Few-shot） | 引导模式 | "例如...这样的输出" |
| 约束条件 | 限制范围 | "只用标准库，不要第三方包" |
| 输出格式 | 规定样式 | "用 Markdown 表格" |

### 实际案例

```python
# 差 Prompt
prompt = "解释机器学习"  
# → 可能得到泛泛的、教科书式的回答

# 好 Prompt
prompt = """你是一个数据仓库工程师，向团队的新人解释机器学习。
要求：
- 从数据库的角度类比
- 控制在 300 字以内
- 包含一个 SQL 类比例子"""
# → 得到针对性、上下文化的回答
```

`demo.py` 中的 `demo_prompt_engineering()` 展示了同一个问题用不同 Prompt 得到的截然不同的回答。

## 总结：LLM 的 8 个概念

```
LLM = 超级"输入法联想"
       ↓
Token = 积木块（不是字，不是词）
       ↓
Context Window = 短期记忆的容量
       ↓
Transformer = 并行处理一切的"工厂流水线"
       ↓
Attention = Ctrl+F 全文搜索
       ↓
幻觉 = 模型在"编故事"（目标不是说真话）
       ↓
长上下文失忆 = 在万人体育场找朋友
       ↓
Prompt = 起点（决定续写方向）
```
