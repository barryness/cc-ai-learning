## 审查流程

### 1. 安全审查（优先级最高）
- [ ] 是否有 SQL 注入风险？
- [ ] 是否有 XSS 风险？
- [ ] 敏感信息是否硬编码（密钥、密码、Token）？
- [ ] 输入校验是否充分（类型、长度、范围）？
- [ ] 认证和授权是否正确实现？

### 2. 逻辑审查
- [ ] 边界条件是否处理（空值、零值、负数）？
- [ ] 错误处理是否完善（不吞噬异常）？
- [ ] 并发场景是否安全（竞态条件）？
- [ ] 事务边界是否正确？

### 3. 性能审查
- [ ] 是否有 N+1 查询？
- [ ] 是否有不必要的循环或重复计算？
- [ ] 大数据量场景是否考虑分页？
- [ ] 是否有可缓存的计算结果？

### 4. 风格审查
- [ ] 命名是否清晰、有意义？
- [ ] 函数是否过长（> 30 行）？
- [ ] 是否有无用的 import 或 dead code？
- [ ] 类型提示是否完整？

## 输出格式

生成 Review Report，按严重程度排序：

```
🔴 Critical（必须修）—— 安全漏洞、逻辑错误
🟡 Warning（建议修）  —— 性能问题、代码异味
🔵 Info（参考）       —— 风格建议、最佳实践
```

每个问题包含：
- **位置**: 文件名:行号
- **问题**: 具体是什么问题
- **建议**: 怎么修改
