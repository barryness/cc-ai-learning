## 1. 什么是 Prompt Engineering

**Prompt Engineering = 用结构化输入控制 LLM 输出质量**

LLM 的本质是"续写机器"——给定前文，预测下一个 Token。Prompt 决定了 LLM "从哪里开始想"，因此直接决定了输出的方向、质量、风格。

**核心认知**：你不是在"命令"LLM，你是在给它设定一个"思考的起点"。起点不同，续写方向完全不同。

```
Prompt 质量 → 激活的知识区域 → 输出质量

差的 Prompt：LLM 瞎猜你要什么 → 泛泛而谈、答非所问
好的 Prompt：LLM 精确知道你是谁、要什么、什么格式 → 精准输出
```

## 2. Prompt 五要素

```
┌─────────────────────────────────────────────────────┐
│                    Prompt 五要素                      │
├──────────┬──────────────────────────────────────────┤
│ Role     │ 你是谁？      →  激活特定知识区域           │
│ Context  │ 背景是什么？   →  限定信息范围               │
│ Constraint│ 不能做什么？  →  排除错误输出               │
│ Example  │ 怎么做？      →  示范期望的模式             │
│ Schema   │ 输出长什么样？ →  控制格式和结构             │
└──────────┴──────────────────────────────────────────┘
```

每个要素回答一个关键问题。五个都用上，输出质量最高。但也要看场景——简单任务不需要全部。

## 3. 要素一：Role（角色）

### 它解决什么问题

LLM 在训练时见过各种领域的文本。如果不指定角色，它在"所有知识区域"里取平均——输出泛泛而谈。

设定角色 = 给 LLM 戴上专业眼镜。它看到的"最可能的下文"会大幅偏向那个领域。

### 实际对比

**无角色**：
> 帮我写一个SQL

LLM 输出：追问需求，给一个通用 SELECT 示例（因为它在猜你要什么）

**有角色**：
> 你是一个资深数据仓库工程师，精通SQL性能优化。帮我写一个SQL。

LLM 输出：直接给数据仓库场景的 SQL（CTE + 聚合 + 维度表 JOIN）

### 角色设计原则

| 原则 | 说明 | 例子 |
|------|------|------|
| 具体不要泛 | "资深 DBA" 好过 "数据库专家" | ✓ "MySQL 8.0 认证 DBA，专注 OLAP 查询优化 10 年" |
| 技能要明确 | 列出关键技能标签 | ✓ "精通窗口函数、索引优化、执行计划分析" |
| 立场要清晰 | 角色决定判断标准 | ✓ "你是甲方数据架构师，要评审乙方提交的 SQL" |

### 角色陷阱

```python
# ❌ 模糊角色
SystemMessage(content="你是一个专家")

# ❌ 冲突角色
SystemMessage(content="你是精益求精的工程师，但速度要快")  # 冲突

# ✅ 精确角色
SystemMessage(content="你是资深 MySQL DBA，10 年 OLAP 经验，精通索引优化和慢查询诊断。你的回答风格：先给结论，再给推理过程。")
```

## 4. 要素二：Context（上下文）

### 它解决什么问题

LLM 不知道你是谁、在做什么项目、有什么数据。不给 Context，它只能猜。猜错的代价 = 输出完全不可用。

### Context 的三层结构

```
1. 环境层：数据库版本、语言、框架
2. 数据层：表名、字段、数据量级
3. 目标层：你要达成什么、给谁用
```

### 实际对比

**无 Context**：
> 帮我写一个SQL

LLM 输出：通用示例，不涉及任何具体表名

**有 Context**：
> 数据库：MySQL 8.0，场景：订单数据仓库
> 表名 orders_dw，字段：order_id, customer_id, product_id, amount, order_date, status

LLM 输出：直接使用 `orders_dw` 表，WHERE 条件用 `status`，聚合用 `amount`

### Context 设计清单

```markdown
## 环境信息（必填）
- 数据库/框架/语言版本：MySQL 8.0
- 数据量级：日增 50 万行
- 硬件/SLA：P99 延迟 < 200ms

## 数据结构（必填）
- 表名：orders_dw
- 关键字段：order_id, customer_id, amount, order_date
- 分区/索引现状：按 order_date 分区，已有 idx_customer

## 目标受众（选填）
- 给谁看：DBA / 业务分析师 / 产品经理
```

## 5. 要素三：Constraint（约束）

### 它解决什么问题

Constraint 是"排除你不想要的"。给 LLM 知道"不能做什么"和给"要做什么"同等重要。

### 约束的类型

| 类型 | 目的 | 例子 |
|------|------|------|
| 语法约束 | 保证兼容性 | "只用标准 SQL，不用 MySQL 方言" |
| 性能约束 | 控制资源消耗 | "必须能用索引，禁止全表扫描" |
| 长度约束 | 控制体积 | "不超过 300 字" |
| 风格约束 | 控制表达 | "用中文解释，关键术语保留英文" |
| 安全约束 | 防止破坏 | "只给 SELECT，不给 DROP/TRUNCATE" |

### 写约束的技巧

```
不要只说"要什么" → 同时说"不要什么"

❌ "写一个高效的 SQL"
✅ "写一个能走索引的 SQL，不要用 SELECT *，禁止子查询超过 2 层"
```

### 约束的优先级暗示

LLM 按出现顺序理解约束（越靠前越重要）：

```markdown
1. 索引必须能用上 ← 最高优先级
2. 用标准 SQL ← 中优先级
3. 不超过 300 字 ← 低优先级（遇冲突时可打破）
```

## 6. 要素四：Example（示例）

### 它解决什么问题

Example 是 Prompt Engineering 中最强大的工具。它比任何描述都有效。

LLM 擅长**模式匹配**。你给它一个例子，它会精确模仿例子中的：
- 注释风格
- 缩进方式
- 命名规范
- 逻辑分解方式

这叫 **In-Context Learning（上下文学习）**——不修改模型参数，仅靠 Prompt 中的例子改变行为。

### Few-shot vs Zero-shot

```
Zero-shot（无示例）：
  只给指令 → LLM 自由发挥 → 质量无法控制

Few-shot（有示例）：
  给 1-3 个例子 → LLM 模仿模式 → 质量稳定可控
```

### 写示例的原则

**原则 1：例子必须正确**
错误的例子 = 教模型做错事。每个例子要经过验证。

**原则 2：例子和任务同构**
例子的输入输出格式要和新任务一致。LLM 会精确模仿你给的格式。

**原则 3：1 个好例子 > 3 个差例子**
例子质量比数量重要。一个精准的例子比三个模糊的强。

**原则 4：覆盖边界情况**
如果任务有边界情况，至少给一个边界情况的例子。

### 示例比较

```sql
-- 好例子：注释清楚、逻辑分层、有性能建议
-- 需求：最近30天各品类销售额Top 10
-- 思路：先按品类汇总，再通过窗口函数排名
SELECT category_id, total_amount, rnk FROM (
    SELECT category_id,
           SUM(amount) AS total_amount,
           ROW_NUMBER() OVER (ORDER BY SUM(amount) DESC) AS rnk
    FROM orders_dw
    WHERE order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 30 DAY)
      AND status = 'completed'
    GROUP BY category_id
) t WHERE rnk <= 10;
-- 索引建议：(order_date, status, category_id) 联合索引
```

这个例子同时示范了：注释风格、子查询分层、窗口函数用法、索引建议格式。

## 7. 要素五：Output Schema（输出格式）

### 它解决什么问题

即使 LLM 输出质量很高，格式不对也无法直接用。Output Schema 精确控制输出结构。

### Schema 的表示方式

**方式 1：描述式**（简单任务）

```
用以下结构回答：
- 需求理解：一句话
- SQL：
- 思路说明：2-3句话
- 索引建议：
```

**方式 2：JSON Schema**（需要解析的场景）

```json
{
  "summary": "需求理解的一句话",
  "sql": "完整的 SQL 语句",
  "explanation": "关键逻辑说明",
  "index_suggestion": "建议的索引"
}
```

**方式 3：Markdown 模板**（给人类看的文档）

```markdown
## 需求
{一句话}

## SQL
```sql
{SQL代码}
```

## 关键逻辑
1. ...
2. ...

## 注意事项
- ...
```

### Schema 设计原则

1. **字段要精确**：不要说"返回 SQL"，要说明是否带注释、是否格式化
2. **不要冲突**：Schema 要求的字段和 Constraint 的"不超过 300 字"可能冲突，需要均衡
3. **预留扩展**：如果后续可能加字段，用 JSON 格式而非固定文本格式

## 8. 组合运用：从差到好的进化路径

### 「帮我写一个SQL」的 4 版进化

下面是真实 API 调用的结果对比。

#### V1：裸提示词（无要素）

```
Prompt: "帮我写一个SQL"
```

LLM 输出（摘要）：
> 你需要补充：要执行什么操作？涉及哪些表？具体业务逻辑是什么？
> 这里给你一个通用的 SELECT 查询示例：SELECT ... FROM users WHERE age > 18 ...

**评价**：LLM 完全在猜。输出 = 追问 + 一个无意义的通用示例。

---

#### V2：+Role

```
Prompt:
  你是一个资深数据仓库工程师，精通SQL。
  帮我写一个SQL。
```

LLM 输出（摘要）：
> 先给您一个数据仓库中常见的月度销售汇总示例（假设有订单表 sales 和日期维度表 dim_date）：
> WITH daily_sales AS (...) SELECT ...

**评价**：角色起作用了。LLM 给出了"数据仓库风格"的 SQL（CTE、维度表 JOIN），但仍然是**猜测的表结构**。

---

#### V3：+Role +Context +Constraint

```
Prompt:
  你是资深数据仓库工程师，精通SQL性能优化。
  上下文：MySQL 8.0，订单数据仓库，表 orders_dw
  字段：order_id, customer_id, product_id, amount, order_date, status
  约束：标准SQL、给索引建议、解释原因、不超过300字
```

LLM 输出（摘要）：
```sql
SELECT customer_id, SUM(amount) AS total_amount
FROM orders_dw
WHERE order_date >= CURDATE() - INTERVAL 90 DAY
GROUP BY customer_id
ORDER BY total_amount DESC;
-- 索引建议：(order_date, customer_id, amount) 复合索引
-- 原因：索引第一列 order_date 快速定位数据，第二列完成分组，第三列覆盖查询
```

**评价**：质变！LLM 用了真实的表名和字段，给了优化建议和索引。**Context + Constraint 产生了可用输出**。

---

#### V4：+Role +Context +Constraint +Example +Schema

```markdown
[在 V3 基础上加入：]

参考示例：
  ```sql
  -- 需求：最近30天各品类销售额Top 10（...完整示例...）
  ```

输出格式：
  - 需求理解：
  - SQL：
  - 思路说明：
  - 索引建议：
```

LLM 输出：
```markdown
### 需求理解
统计最近一年每月已完成订单的销售额，找出Top10。

### SQL
[格式化SQL，含注释，含 ROW_NUMBER() 窗口函数]

### 思路说明
1. 筛选最近一年已完成的订单
2. 按年月分组汇总
3. 窗口函数排名、取前10
4. 外层按月份排序

### 索引建议
(order_date, status) 覆盖 WHERE，后续按月份聚合
```

**评价**：输出结构严整、代码有注释、逻辑分层清楚。**Example 教了风格，Schema 锁定了结构**。

---

### 进化总结

```
V1（无要素）  → 答非所问，LLM 在瞎猜
V2（+Role）   → 给了方向，但内容仍通用
V3（+Ctx+Cst）→ 可用！针对性输出，真实表结构
V4（+Ex+Sch） → 专业！风格一致、结构清晰、可直接使用
```

**每加一个要素，输出质量升一级。**

## 9. 常见错误模式

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| Role 太泛 | 专家建议变成通用建议 | Role = 角色 + 技能 + 经验年限 |
| Context 缺失 | LLM 编造不存在的信息 | 给表名、字段、数据量级 |
| Constraint 只说"要什么" | 输出包含不该有的内容 | 同时说"不要什么" |
| Example 太差 | 教会模型做错事 | Example 先自己验证一遍 |
| Schema 和 Constraint 冲突 | LLM 左右为难，输出质量下降 | "不超300字"和"包含完整SQL+解释"冲突时，放宽字数 |
| 所有要素堆一起 | Prompt 太长，LLM 混淆优先级 | 按重要程度排序，次要约束后置 |

## 10. Prompt 模板库

### 代码生成模板

```markdown
你是 {角色}，{经验}，擅长 {技能标签}。

## 上下文
- 项目类型：{描述}
- 技术栈：{语言/框架/版本}
- 数据/代码结构：{关键信息}

## 要求
1. {约束1}
2. {约束2}
3. {约束3}

## 参考示例
{好的输出示例}

## 输出格式
{结构要求}
```

### Bug 诊断模板

```markdown
你是 {角色}，专门排查 {问题类型}。

## 问题描述
{现象 + 报错信息 + 复现步骤}

## 环境信息
{版本、配置、日志片段}

## 排查要求
1. 先定位根因，再给方案
2. 按优先级排列（安全 > 性能 > 可读性）
3. 给出预防措施
```

### 学习/解释模板

```markdown
你是 {老师角色}。

## 听众背景
{技术背景描述}

## 要解释的概念
{概念}

## 要求
1. 从 {类比维度} 类比
2. 用 {听众已有的知识} 作为桥梁
3. 控制在 {字数} 以内
4. 包含一个可运行的代码示例
```

```
R - Role      "你是谁"
C - Context   "什么情况"
C - Constraint "不能怎样"
E - Example   "像这样写"
S - Schema    "长这样"

记法：R C C E S → "Ricky's CES" → 里克的质量检验
```
