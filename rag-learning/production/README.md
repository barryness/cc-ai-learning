# 生产级 RAG 架构 Demo

## 它是什么

`advanced_rag.py` 是 `demo/minimal_rag.py` 的工业级版本——把最小 Demo 里的每个组件都替换成了生产环境的标准实现。

## 升级点 × 对应的练习问题

每个升级都不是凭空设计的，而是你在 8 个练习中真实遇到过的问题：

| 升级点 | minimal_rag.py | advanced_rag.py | 练习中踩的坑 |
|--------|---------------|-----------------|-------------|
| 文档分块 | 整篇文档 | 语义分块 (500字/块 + 50字重叠) | 练习2：长文档中关键信息被稀释，top_k 难以兼顾 |
| 检索算法 | TF-IDF + jieba | Embedding 模型 (sentence-transformers) | 练习3：字面匹配不理解语义，"办公地点" vs "办公地址"得分低 |
| 存储方式 | Python 内存列表 | ChromaDB 向量数据库 | 练习5：文件多了内存扛不住，需要持久化和索引加速 |
| 检索策略 | 纯向量检索 | 向量 + BM25 + RRF 混合检索 | 练习6：精确词（"Claude code"）向量检索召回不足，需关键词互补 |
| 分词依赖 | jieba（手动分词） | Embedding 模型自带 tokenizer | 练习1&7：jieba 词典不全，"年假"被拆成"年"+"假" |
| 日志 | print() | logging 模块 | — |

### 为什么 Embedding 模型能解决中文分词问题？

```
TF-IDF + jieba:
  "年假" → jieba 词典里没有 → ["年", "假"] → 检索信号被稀释

Embedding 模型:
  "年假" → 直接编码成向量 [0.23, -0.15, 0.67, ...]
  → 不依赖分词词典，模型自己理解"年假"的语义
  → 和"休假""假期"等词在向量空间里很接近
```

### 为什么需要分块？

```
不分块（minimal_rag）：
  文档 "公司介绍" (500+ 字) → 1 个向量 → "年假" 信号被其他 490 字淹没

分块后（advanced_rag）：
  文档 "办公政策" → Chunk 1 "混合办公制度..." Chunk 2 "年假15天..."
  → "年假" 在 Chunk 2 里占比高 → 检索精准命中
```

## 架构全景

```
  ┌─────────────┐    ┌──────────────┐    ┌───────────────┐
  │  PDF/网页/   │───▶│  文档分块器    │───▶│  Embedding    │
  │  数据库...   │    │  (Chunker)    │    │  (Encoder)    │
  └─────────────┘    └──────────────┘    └───────┬───────┘
                                                  │
                  ┌───────────────────────────────▼────────────────────┐
                  │               向量数据库 (ChromaDB)                  │
                  └───────────────────────┬────────────────────────────┘
                                          │
  ┌──────────┐                    ┌───────▼────────┐
  │  用户问题  │──────────────────▶│   混合检索器     │
  └──────────┘                    │ 向量 + BM25     │
                                  │   RRF 融合      │
                                  └───────┬────────┘
                                          │ top_k 文档
                                  ┌───────▼────────┐
                                  │   Prompt 构造   │
                                  │   拼接上下文     │
                                  └───────┬────────┘
                                          │
                                  ┌───────▼────────┐
                                  │     LLM        │
                                  │   生成答案      │
                                  └────────────────┘
```

## 运行

```bash
cp ../demo/.env.example .env  # 填入 API Key
pip install -r requirements.txt
python3 advanced_rag.py
```

首次运行会下载 sentence-transformers 模型（约 80MB）。

## 和 Demo 的关系

```
minimal_rag.py  ──► 理解"RAG 是什么"，用最简单的工具
                         │
                    8 个练习 ──► 踩坑、理解每个组件为什么需要升级
                         │
advanced_rag.py  ──► 知道"升级后怎么做"，每个组件都有对应
```

学完 RAG 再回来看这个文件，你应该能看懂每一行在解决什么问题。
