# 练习7：RAG 效果评估

## 评估方法

三种指标，从严格到宽松：

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| ROUGE-L | 最长公共子序列 | 快速、可复现 | 只看字面，语义可能对但得分低 |
| 关键词匹配 | 共同词占比 | 简单直观 | 忽略词序和否定词 |
| LLM 打分 | 让 LLM 判断准确性 | 最接近人类判断 | 花钱、结果不完全可复现 |

## 评估数据集

10 个问答对，分三类：

| 类型 | 数量 | 说明 |
|------|------|------|

| 简单事实 | 2 题 | 一句话就能回答 |
| 知识库外 | 1 题 | 测试拒绝回答的能力 |

## 踩坑发现：jieba 分词导致评估失真

#6 翻车的原因：

```
查询 "公司年假有多少天？"
jieba 分词: ['公司', '年', '假', '有', '多少', '天', '？']
                                      ↑    ↑
                            拆成了两个常见字，关键信息被稀释

正确的分词: ['公司', '年假', '有', '多少', '天', '？']
                      ↑
                   "年假"应该是一个词
```

这是 jieba 默认词典的覆盖问题。修复方法：

```python
jieba.add_word('年假')  # 告诉 jieba 这是一个词
```

**这个发现很重要**：评估的价值不在于"打分"本身，而在于**发现哪些查询检索失败了**。ROUGE-L 0.182 是一个异常信号，引导我们发现了一个分词问题——这和第一次运行 RAG 时遇到的"中文分词"问题是同一个根因的不同表现形式。

## 评估在 RAG 开发中的定位

```
开发阶段：跑几个问题，人眼看 → 快速迭代
评估阶段：10-100 个标准 QA 对 + ROUGE-L → 发现异常
上线前：  1000+ QA 对 + LLM 打分 + 人工抽检 → 质量把关
上线后：  用户反馈（点赞/踩）+ A/B 测试 → 持续优化
```

评估不是做一次就完了，而是融入开发流程的持续过程。
