# 向量数据库对比指南

## 一、为什么需要向量数据库？

### 核心问题：传统数据库无法理解"语义"

传统数据库擅长**精确匹配**：

```sql
SELECT * FROM products WHERE name = 'iPhone 15';
```

但无法处理**语义查询**：

> "找一款适合拍照的手机，预算5000左右"

这种查询的核心不是关键词匹配，而是**语义相似度**——传统数据库完全无法做到。

### 向量的本质

向量是一串数字（如 `[0.23, -0.15, 0.78, ...]`），它将文本、图片、音频的"语义"编码到高维空间中。语义相近的内容，向量距离就相近。

```
"苹果手机"  →  [0.12, 0.85, -0.33, ...]
"iPhone"    →  [0.11, 0.83, -0.31, ...]   ← 距离很近！
"天气预报"  →  [-0.67, -0.21, 0.54, ...]  ← 距离很远
```

### 三大核心场景

| 场景 | 说明 | 例子 |
|------|------|------|
| **语义搜索** | 按意思搜索，不是按关键词 | "适合拍照的手机" |
| **RAG（检索增强生成）** | LLM 回答前先检索相关知识 | 企业知识库问答 |
| **相似推荐** | 找到语义最接近的内容 | "看了这个还看了..." |

## 二、与传统数据库的本质区别

| 维度 | 传统数据库（SQL） | 向量数据库 |
|------|-------------------|------------|
| **查询方式** | 精确匹配（`WHERE name = 'x'`） | 近似匹配（`ORDER BY distance LIMIT k`） |
| **索引机制** | B-Tree / Hash / 倒排索引 | ANN 索引（HNSW / IVF / PQ） |
| **数据单元** | 结构化行/列 | 高维向量（128~4096 维） |
| **结果确定性** | 确定（相同查询 = 相同结果） | 近似（ANN 允许精度换速度） |
| **核心操作** | CRUD + JOIN + 聚合 | 向量相似度计算（Cosine / Euclidean） |
| **扩展方式** | 垂直扩展（更强 CPU/磁盘） | 水平扩展（更多节点并行检索） |

**关键认知**：向量数据库不是要替代传统数据库，而是补充——**精确查询用 SQL 数据库，语义查询用向量数据库**。

### 3.2 各系统详解

#### MySQL

```
定位：最流行的关系型数据库
向量：不支持（也没有计划支持）
```

- **优势**：生态成熟、运维简单、ACID 事务
- **劣势**：完全无法处理向量，只能存向量为 BLOB，查询靠全表扫描
- **适合场景**：用户信息、订单、库存等结构化数据
- **不适合**：任何语义搜索、RAG

#### PostgreSQL + pgvector

```
定位：关系型数据库 + 向量扩展
向量：pgvector 插件，支持 IVFFlat / HNSW 索引
```

- **优势**：一个数据库同时做 OLTP + 向量检索，已有 PG 的用户零迁移成本
- **劣势**：百万级以上向量性能不如专用向量数据库；pgvector 索引构建慢
- **适合场景**：中小规模 RAG（百万文档以内），团队已有 PostgreSQL 使用经验
- **与专用向量库的性能差距**：10万向量以内差不多；百万级以上，Milvus 快 3-10 倍

#### Elasticsearch

```
定位：全文搜索引擎
向量：8.0+ 支持 dense_vector + KNN
```

- **优势**：倒排索引 + 向量检索可以混合查询（Hybrid Search），过滤能力极强
- **劣势**：向量检索是"后加的"，内存占用大，查询延迟不如专用向量库稳定
- **适合场景**：需要"全文 + 语义"混合搜索的场景（如电商搜索"红色连衣裙"）
- **核心差异**：ES 的核心是倒排索引（关键词），向量是辅助；Chroma/Milvus 的核心就是向量

#### Chroma

```
定位：AI 原生向量数据库，面向开发者
向量：HNSW 索引，内置 embedding 函数
```

- **优势**：API 最简洁，与 LangChain/LlamaIndex 深度集成，内置多种 embedding 模型
- **劣势**：单机部署，不适合超大规模（亿级+）；社区相对年轻
- **适合场景**：原型开发、学习、小规模 RAG 应用（十万级文档）
- **一句话**：向量数据库界的 SQLite——轻量、内嵌、零配置

#### FAISS

```
定位：Meta 开源的向量相似度检索库
向量：纯 C++ 核心，Python 接口，支持 GPU
```

- **优势**：速度极致（纯 C++，支持 GPU），索引类型丰富（10+ 种），十亿级向量毫秒查询
- **劣势**：它不是数据库——没有持久化、没有 CRUD、没有 SQL、没有分布式。所有数据在内存中
- **适合场景**：离线批量检索、学术研究、作为其他向量数据库的底层引擎
- **关键认知**：FAISS 是"库"不是"数据库"——你需要自己管理数据存储和分布式

#### Milvus

```
定位：云原生分布式向量数据库
向量：支持 10+ 种索引类型（HNSW / IVF / DiskANN 等）
```

- **优势**：分布式架构、十亿级可扩展、混合查询（标量 + 向量）、多租户隔离
- **劣势**：部署复杂（依赖 etcd、MinIO、Pulsar/Kafka）、资源消耗大、学习曲线陡
- **适合场景**：生产级大规模 RAG、以图搜图、推荐系统
- **一句话**：向量数据库界的 PostgreSQL——功能完整、生产就绪、运维需要投入

## 四、选型决策树

```
要选向量数据库？
│
├─ 只是学习/Demo/原型？
│   └─ → Chroma（最轻量，pip install 即可）
│
├─ 已有 PostgreSQL？
│   └─ 数据量 < 百万 → pgvector（零迁移成本）
│
├─ 需要全文 + 语义混合搜索？
│   └─ → Elasticsearch（唯一的混合搜索王者）
│
├─ 生产环境、大规模、低延迟？
│   └─ → Milvus（分布式、十亿级）
│
├─ 离线分析、学术研究、极致性能？
│   └─ → FAISS（但需要自己搭持久化和分布式）
│
└─ 纯结构化查询、不需要语义？
    └─ → MySQL / PostgreSQL（传统的就够了）
```

## 五、关键术语速查

| 术语 | 含义 | 类比 |
|------|------|------|
| **Embedding** | 将内容转为高维向量 | 给文本拍一张"语义照片" |
| **ANN** | 近似最近邻搜索 | 不找最像的那个人，找"足够像"的前10个 |
| **HNSW** | 分层可导航小世界图（一种 ANN 索引） | 高速公路 → 主干道 → 小巷的导航层次 |
| **Cosine Similarity** | 余弦相似度（最常用的向量距离） | 看两个箭头的方向是否一致 |
| **Top-K** | 返回最相似的 K 个结果 | 搜索结果的 Top 10 |
| **Chunk** | 将长文档切分成小块 | 一本书拆成段落来检索 |
