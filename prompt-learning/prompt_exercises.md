# Prompt Engineering 练习题

> 10 道练习题，覆盖 Role、Context、Constraint、Example、Output Schema 的渐进式训练。
> 建议用 `llm_playground.py` 的实验4（Prompt 实验室）来完成，可以直接看到不同 Prompt 的效果差异。

---

## 练习 1：Role 的威力 —— 同一个问题，四个角色

### 背景
Role 决定了 LLM "戴什么眼镜"看你的问题。同样的输入，不同角色产生完全不同的输出。

### 任务
用以下 4 个角色分别问同一个问题，比较输出差异。

**问题**：解释什么是「数据库死锁」

**角色**：
1. 无角色（裸问）
2. 幼儿园老师（给 5 岁小孩讲）
3. 资深 DBA（给团队做技术分享）
4. 产品经理（从用户体验角度）

### 输出要求
记录每个回答的：篇幅、用词难度、信息结构、是否有趣。

### 思考
哪个角色更"有用"取决于什么？是不是越专业越好？

## 练习 2：Context 的精确度 —— 从模糊到清晰

### 背景
Context 是 LLM 的地图。地图越详细，导航越准确。

### 任务
分 3 个版本，逐步添加 Context，看 LLM 如何从"瞎猜"到"精准"。

**问题**：写一个查询报表的 SQL

| 版本 | Context 内容 |
|------|-------------|
| V1 | 无 |
| V2 | "MySQL，订单表，查销售额" |
| V3 | "MySQL 8.0。订单表 orders（order_id, customer_id, amount, create_time, status）。需求：按周统计已完成订单的销售额，取最近 12 周，横轴是周一到周日，纵轴是销售额" |

### 输出要求
记录每版输出的：是否用了真实表名、是否有索引建议、是否可直接运行。

### 思考
V2 和 V3 之间发生了什么？Context 的哪条信息导致了质变？

## 练习 3：Constraint 的取舍 —— 当需求冲突时

### 背景
不是所有约束都应该遵守。有些约束互相冲突，需要你判断优先级。

### 任务
设计一个 Prompt，要求 LLM 给出一段代码的 Code Review。

**代码**：一个 500 行的 Python 函数，不做拆分。

**冲突约束**：
- 约束 A：详细到逐行的 Review
- 约束 B：总输出不超过 200 字

### 输出要求
测试以下三种写法，观察 LLM 的行为：
1. 两个约束都写
2. 只写约束 A
3. 约束 A + 强调"如果不能两全，优先详细 Review"

### 思考
LLM 如何处理冲突约束？它倾向于遵守哪个？

## 练习 4：Example 的魔力 —— 1 个例子 vs 0 个例子

### 背景
Example 是最被低估的 Prompt 要素。一个正确的例子胜过 100 行的描述。

### 任务
设计一个 Prompt，让 LLM 生成 Git Commit Message。

**场景**：修改了 3 个文件：`auth.py`（修复 token 过期 bug）、`config.py`（新增 timeout 配置）、`test_auth.py`（新增测试用例）。

**对比**：
1. Zero-shot：只描述规则，不给例子
2. One-shot：给 1 个规范的 commit message 示例
3. Three-shot：给 3 个不同场景的示例

### 输出要求
记录三版的格式一致性、动词选择、详细程度。

### 思考
1 个例子和 3 个例子差距大吗？差在哪？

## 练习 5：Output Schema —— 自由格式 vs 结构化

### 背景
如果你的输出要传给另一个程序，自由格式 = 不可用。

### 任务
让 LLM 从一段需求描述中提取「数据仓库表设计」。

**需求描述**：
> 我们要一个订单分析系统。需要记录谁在什么时候买了什么、花了多少钱、订单状态是什么。另外还需要知道商品属于哪个品类，以及这个品类是哪个事业部负责的。对了还有退款和退货也需要考虑。

**对比**：
1. 无 Schema："帮我设计表"
2. JSON Schema：要求输出 JSON（字段：tables[].name, tables[].columns[].name, tables[].columns[].type, tables[].relationships）
3. Markdown 表格 Schema

### 输出要求
评价每版的可解析性、完整性、是否遗漏了退款/退货。

### 思考
Schema 是否影响了 LLM 的"思考深度"？（即：结构化要求是否促使 LLM 更细致地分析？）

## 练习 6：诊断差 Prompt —— 找出问题并优化

### 背景
Prompt Engineering 不只是"写好"，更是"诊断差"。

### 任务
以下 Prompt 有什么问题？逐一诊断并改写。

```markdown
你是一个专家。

帮我写一份数据仓库的设计文档。

要详细、要全面、要专业。

不要写得太复杂，新人也能看懂。
```

### 诊断维度
- Role 是否足够具体？
- Context 是否缺失？
- Constraint 是否有冲突？
- 是否缺少 Example 和 Schema？

### 输出要求
写出你的改进版 Prompt，并解释每一项改了什么、为什么。

## 练习 7：渐进式 Prompt 设计 —— 一个需求的 4 次迭代

### 背景
好的 Prompt 不是一次写出来的，是迭代出来的。

### 任务
需求：**"让 LLM 帮我写一段 Python，从 Hive 导出数据到 CSV"**

要求你设计 4 个版本的 Prompt：
1. **V1**：裸需求
2. **V2**：添加 Role + Context（技术栈、表结构、数据量）
3. **V3**：添加 Constraint（性能、错误处理、日志）
4. **V4**：添加 Example + Output Schema

### 输出要求
- 每个版本标注用了哪些要素
- 实际运行，记录每版输出
- 写一段总结：每加一个要素，输出改变了什么

### 提示
```
Hive 表：dwd.order_detail_di（日增量，1000 万行/天）
字段：order_id, product_id, amount, dt
导出目标：CSV，按 dt 分区，每个分区一个文件
```

## 练习 8：跨领域 Prompt —— SQL 到 Python 到文档

### 背景
同一个需求场景，不同输出形式需要不同的 Prompt 设计。

### 任务
需求：**"订单数据仓库，计算各品类复购率"**

分别设计 3 个 Prompt，要求 LLM 输出：
1. **SQL 版本**：可运行的查询语句
2. **Python 版本**：用 PySpark 的实现
3. **文档版本**：给产品经理看的业务说明

### 输出要求
每个 Prompt 必须包含 Role + Context + Constraint + Output Schema。
Context 部分可以复用，Role/Constraint/Schema 每个版本不同。

### 思考
三个 Prompt 最大的不同在哪？哪些要素是共用的？

## 练习 9：复杂约束 —— 数据安全场景

### 背景
安全相关场景中，Constraint 的设计是核心。错一个约束可能造成线上事故。

### 任务
设计一个 Prompt，让 LLM 生成一段数据库迁移脚本。

**需求**：orders 表新增 `refund_amount` 字段（DECIMAL(12,2)，默认 0.00，不可为 NULL）。

**安全约束**：
1. 必须有回滚脚本
2. 不能锁表超过 5 秒（2000 万行表）
3. 必须先检查字段是否已存在（幂等）
4. 新字段不允许为 NULL（防止计算错误）
5. 迁移前先备份（注释中写明备份命令）

### 输出要求
写出你的 Prompt，并评价 LLM 的输出是否满足了所有 5 条安全约束。

### 思考
如果 LLM 忽略了某个约束，你该怎么调整 Prompt——直接强调那个约束？还是调整约束的顺序？

## 练习 10：组合实战 —— 设计一个「DWH 助手」System Prompt

### 背景
综合运用所有五个要素，设计一个生产级的 System Prompt。

### 需求
你的团队需要一个「数据仓库开发助手」，日常用到的场景：
- 写 SQL（各种复杂度）
- 诊断慢查询
- 设计表结构
- 做数据质量检查

### 要求
设计一个通用的 System Prompt，要求：
1. **Role**：明确角色定位和技能
2. **Context**：团队的技术栈（MySQL + Hive + Spark）
3. **Constraint**：至少 5 条行为准则（安全 > 性能 > 可读性 > 简洁 > 创新）
4. **Example**：给 2 个"好的回答"示例（一个 SQL 场景 + 一个诊断场景）
5. **Schema**：根据问题类型自动选择合适的输出结构

### 评估标准

| 标准 | 说明 |
|------|------|
| 角色一致性 | 回答风格是否始终体现 DWH 工程师特质 |
| SQL 质量 | 生成的 SQL 是否有注释、有索引建议、考虑性能 |
| 安全边界 | 对 DROP/TRUNCATE 等危险操作是否拒绝 |
| 诚实度 | 不知道时是否承认而非编造 |
| 结构清晰 | 输出是否有清晰的信息层级 |

### 测试用例
用以下 3 个问题测试你的 Prompt：

```python
test_cases = [
    "帮我创建一个订单表",                     # 应该设计字段、索引、分区
    "为什么这个 SQL 执行很慢？SELECT ...",   # 应该分析执行计划
    "把所有订单数据删掉重建",                # 应该拒绝并警告
]
```

### 输出要求
- 提交你的完整 System Prompt
- 附上 3 个测试用例的实际输出
- 自评：哪些方面做得好？哪些可以改进？

## 答案要点

<details>
<summary>练习 1 要点</summary>

- 无角色：教科书定义，200-300 字，术语密集
- 幼儿园老师：比喻"两个小朋友抢一个玩具"，50-100 字，零术语
- 资深 DBA：从事务隔离级别讲，包含 MySQL 真实案例
- 产品经理：从"用户操作卡住"讲，关注体验而非技术

**结论**：角色决定的是"知识的层次"，而非"知识的正确性"。
</details>

<details>
<summary>练习 3 要点</summary>

LLM 倾向于遵守靠后的明确指令。如果两个约束冲突，通常遵守：
- 出现在 Constraint 列表更后面的
- 明确标注"优先"的

所以"详细 Review" + "优先详细 Review"效果最好。
</details>

<details>
<summary>练习 6 诊断结果</summary>

- Role："一个专家"太泛 → 需指定领域
- Context：完全缺失，不知道要设计什么规模的仓库
- Constraint：内部冲突——"详细专业"和"新人能看懂"在无额外指导时互斥
- Example：缺失
- Schema：缺失

改进方向：具体化每个要素，用 Example 示范"专业但易懂"是什么样。
</details>

<details>
<summary>练习 10 设计提示</summary>

关键设计点：
1. 危险操作拒绝：把"拒绝 DROP/TRUNCATE/DELETE 无 WHERE"放在 Constraint 第一条
2. 不知道就承认：比编造一个"看起来对"的 SQL 安全得多
3. 2 个 Example 覆盖不同场景（生成 + 诊断）
4. Schema 用 if-else 逻辑：如果是 SQL 请求则输出 ABC，如果是诊断则输出 XYZ
5. 优先级的显式声明："安全性 > 正确性 > 性能 > 可读性"
</details>
